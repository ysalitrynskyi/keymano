# Contributing to Keymano

Thanks for your interest in improving Keymano. This guide covers how to build,
test, and submit changes.

## Ground rules

- Be respectful — see the [Code of Conduct](CODE_OF_CONDUCT.md).
- Keep the Rust core (`crates/keylayout-core`) pure and Tauri-free.
- All user-facing strings go through i18n (`src/locales`), never hardcoded JSX.
- Add or update tests with your change; keep CI green.

## Project layout

```
crates/keylayout-core   pure Rust: model, parse, serialize, modifiers, resolve,
                        validate, templates, ids, special_keys, bundle
crates/keymano-session  Tauri-free document session + undo/redo
src-tauri               thin Tauri command shell over the session
src                     React + TS frontend (keyboard, pages, ipc, i18n)
docs/                   GETTING_STARTED, DEVELOP, i18n READMEs, screenshots
```

Read [`docs/DEVELOP.md`](docs/DEVELOP.md) before a substantial change.

## Develop

```bash
pnpm install
pnpm dev          # UI in a browser via the web-mock (no Rust) — fastest loop
pnpm tauri dev    # full desktop app (needs Rust + a system WebView)
```

## Test & lint (run before opening a PR)

```bash
pnpm lint                                          # eslint
pnpm exec tsc -b                                   # typecheck
pnpm test                                          # frontend (vitest)
cargo test -p keylayout-core -p keymano-session    # Rust
cargo clippy --workspace --all-targets -- -D warnings
cargo fmt --all --check
```

Everything except the native desktop build also runs in Docker — see the
[README](README.md#run-with-docker).

## Pull requests

1. Fork and branch from `main` (`feat/…`, `fix/…`, `docs/…`).
2. Make focused commits; follow [Conventional Commits](https://www.conventionalcommits.org/)
   (`feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:`).
3. Ensure the checks above pass locally.
4. Open a PR describing the change and how you tested it. Link any related issue.
5. For new locales or string changes, keep the locale key sets in parity (the
   `locales.test.ts` test enforces this).

## Releasing

Releases are fully automated by CI. To cut one:

1. Bump `version` in `package.json` (the Tauri bundles read it) and update
   `CHANGELOG.md`.
2. Commit, then push a matching `v*` tag:
   ```bash
   git tag v0.1.0 && git push origin v0.1.0
   ```
3. On that tag, CI:
   - builds + pushes the web image to GHCR (`:0.1.0`, `:0.1`, `:latest`);
   - builds the desktop bundles — macOS (universal `.dmg`), Windows
     (`.msi` / `.exe`), Linux (`.deb` / `.AppImage`);
   - creates a GitHub Release (curated description + autogenerated notes) with
     every bundle attached, then publishes it.

Tag the version `package.json` declares. A tag containing `-`
(e.g. `v0.1.0-rc1`) is published as a prerelease.

## Reporting bugs / requesting features

Use the GitHub issue templates. For security issues, **do not** open a public
issue — see [SECURITY.md](SECURITY.md).
